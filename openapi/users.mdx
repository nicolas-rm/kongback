## Base

Base path: `/api/users`

Todos los endpoints requieren autenticacion y el permiso indicado en cada operacion. Las mutaciones aceptan JSON y las respuestas no exponen `passwordHash`, tokens, secretos 2FA ni recovery codes.

Esta API es mixta:

- Sin `X-Company-Id`, los endpoints requieren permisos desde un acceso global (`companyId: null`) y operan como administracion de sistema.
- Con `X-Company-Id`, los endpoints mixtos operan dentro de la compania seleccionada. El backend valida que la compania exista, este activa y que el usuario autenticado tenga acceso a ella.
- En contexto de compania, los listados y lecturas solo devuelven usuarios con acceso directo a esa compania. No mezclan usuarios de otras companias.
- En contexto de compania, asignar/reemplazar/eliminar accesos solo afecta accesos de esa compania. No permite crear accesos globales ni accesos para otra compania.
- Si el acceso del usuario autenticado esta limitado con `scopeKey: "subCompanyId"`, los endpoints mixtos solo muestran y administran accesos de esas subcompanias.
- Crear usuarios, editar datos base, eliminar usuarios, cambiar contrasena, desvincular 2FA y reenviar credenciales siguen siendo operaciones globales. Al crear usuario se puede enviar un acceso inicial opcional para relacionarlo con una compania desde el alta.

## Modelos

### User

```json
{
    "id": "9f5f0f4a-8f88-4f0b-9a18-6eb3f1e9b7a1",
    "username": "admin",
    "email": "admin@example.com",
    "fullName": "Admin User",
    "preferredLanguage": "es",
    "status": "active",
    "mustChangePassword": false,
    "emailVerified": true,
    "twoFactorEnabled": false
}
```

| Campo                | Tipo                              | Descripcion                                                                  |
| -------------------- | --------------------------------- | ---------------------------------------------------------------------------- |
| `id`                 | `string`                          | UUID v4 del usuario.                                                         |
| `username`           | `string`                          | Nombre de usuario unico.                                                     |
| `email`              | `string`                          | Email unico; se normaliza a minusculas en create/update.                     |
| `fullName`           | `string`                          | Nombre completo.                                                             |
| `preferredLanguage`  | `string`                          | Idioma preferido; por defecto `es`.                                          |
| `status`             | `active \| inactive \| suspended` | Estado operativo del usuario.                                                |
| `mustChangePassword` | `boolean`                         | Indica si debe cambiar la contrasena en su siguiente flujo de autenticacion. |
| `emailVerified`      | `boolean`                         | Derivado de `emailVerifiedAt`.                                               |
| `twoFactorEnabled`   | `boolean`                         | Indica si el usuario tiene 2FA activo.                                       |

## Crear usuario

`POST /api/users`

Requiere `users.create`.

### Body

```json
{
    "username": "admin",
    "email": "admin@example.com",
    "fullName": "Admin User",
    "status": "active",
    "access": {
        "roleId": "768684d0-5892-442c-b52f-ad7d0ef372c6",
        "companyId": "c118d5f0-55c7-48f0-a17f-784909a61f97"
    }
}
```

| Campo               | Tipo                              | Requerido | Reglas                                                                                                                     |
| ------------------- | --------------------------------- | --------- | -------------------------------------------------------------------------------------------------------------------------- |
| `username`          | `string`                          | Si        | Se recorta y colapsa espacios.                                                                                             |
| `email`             | `string`                          | Si        | Se recorta, colapsa espacios y normaliza a minusculas.                                                                     |
| `fullName`          | `string`                          | Si        | Se recorta y colapsa espacios.                                                                                             |
| `password`          | `string`                          | No        | Minimo 8 caracteres, una minuscula, una mayuscula y un numero. Si se omite, backend genera una contrasena temporal segura. |
| `status`            | `active \| inactive \| suspended` | No        | Si se omite, usa `active`.                                                                                                 |
| `preferredLanguage` | `string`                          | No        | Se normaliza a minusculas; si se omite, usa `es`.                                                                          |
| `access`            | `object`                          | No        | Acceso inicial del usuario. Usa el mismo contrato de asignacion de acceso.                                                 |

Si `access` se envia, el backend crea el usuario y el acceso en una sola transaccion. `access.roleId` es requerido dentro de ese objeto. `access.companyId` puede ser `null` para acceso global o el UUID de una compania activa. Para limitarlo a una subcompania, usa `scopeKey: "subCompanyId"` y `scopeId` con el UUID de una subcompania activa perteneciente a esa compania.

Si `password` se omite, el usuario queda con `mustChangePassword: true` y el backend envia correo de bienvenida con credenciales temporales usando `preferredLanguage` o `es` por defecto.

El front debe enviar roles, no permisos individuales. Los permisos efectivos salen de los roles asignados.

### Response `201`

Devuelve `User`.

## Listar usuarios

`GET /api/users`

Requiere `users.read-list`.

Si se envia `X-Company-Id`, lista solo usuarios con acceso directo a esa compania. Si se omite, requiere acceso global y lista usuarios del sistema.

### Query params

| Parametro | Tipo                              | Reglas               | Descripcion                                                                          |
| --------- | --------------------------------- | -------------------- | ------------------------------------------------------------------------------------ |
| `page`    | `number`                          | Entero >= 1          | Pagina solicitada; por defecto `1`.                                                  |
| `limit`   | `number`                          | Entero entre 1 y 100 | Registros por pagina; por defecto `10`.                                              |
| `search`  | `string`                          | Opcional             | Busca por `username`, `email` o `fullName`, sin distinguir mayusculas.               |
| `all`     | `boolean`                         | Opcional             | Si es `true`, omite el offset y solicita hasta 100 registros si no se envia `limit`. |
| `status`  | `active \| inactive \| suspended` | Opcional             | Filtra por estado.                                                                   |

### Example

```http
GET /api/users?page=1&limit=20&search=admin&status=active
```

```http
GET /api/users?page=1&limit=20
X-Company-Id: c118d5f0-55c7-48f0-a17f-784909a61f97
```

### Response `200`

Devuelve `PaginatedUserResponse`.

## Obtener usuario

`GET /api/users/:id`

Requiere `users.read-one`.

Si se envia `X-Company-Id`, el usuario solicitado debe tener acceso directo a esa compania; si no, devuelve `null`.

### Path params

| Parametro | Tipo   | Descripcion     |
| --------- | ------ | --------------- |
| `id`      | `uuid` | ID del usuario. |

### Response `200`

Devuelve `User`. Comportamiento actual: si el usuario no existe, devuelve `null`.

## Listar accesos

`GET /api/users/:id/access`

Requiere `users.access.read`.

Devuelve los accesos activos del usuario. El usuario debe existir y estar en estado `active`.

Si se envia `X-Company-Id`, devuelve solo accesos activos de esa compania. Si se omite, requiere acceso global y devuelve todos los accesos activos.

### Response `200`

```json
[
    {
        "id": "2db0b73a-b3cb-49a6-8fdf-f81789ab2117",
        "companyId": "c118d5f0-55c7-48f0-a17f-784909a61f97",
        "scopeKey": null,
        "scopeId": null,
        "role": {
            "id": "768684d0-5892-442c-b52f-ad7d0ef372c6",
            "code": "company.admin",
            "name": "Company admin"
        }
    }
]
```

## Asignar acceso

`POST /api/users/:id/access`

Requiere `users.access.assign`.

El usuario debe estar `active`. El rol debe existir. Si `companyId` se envia, la compania debe existir y estar `active`.

Si se envia `X-Company-Id`, el acceso queda forzado a esa compania. En ese modo `companyId` puede omitirse en el body; si se envia, debe coincidir con `X-Company-Id`.

### Body

```json
{
    "roleId": "768684d0-5892-442c-b52f-ad7d0ef372c6",
    "companyId": "c118d5f0-55c7-48f0-a17f-784909a61f97"
}
```

| Campo       | Tipo             | Requerido | Descripcion                                                                              |
| ----------- | ---------------- | --------- | ---------------------------------------------------------------------------------------- |
| `roleId`    | `uuid`           | Si        | Rol a asignar.                                                                           |
| `companyId` | `uuid \| null`   | No        | Compania especifica del acceso. String vacio se transforma a `null`.                     |
| `scopeKey`  | `string \| null` | No        | Scope generico opcional para reglas futuras. String vacio se transforma a `null`.        |
| `scopeId`   | `string \| null` | No        | ID del scope generico opcional para reglas futuras. String vacio se transforma a `null`. |

Reglas de alcance:

- `companyId: null`: acceso global.
- `companyId` con `scopeKey: null` y `scopeId: null`: acceso a toda esa compania.
- `companyId` con `scopeKey: "subCompanyId"` y `scopeId: <subCompanyId>`: acceso limitado a esa subcompania. La subcompania debe estar activa y pertenecer a `companyId`.

Si el request envia `X-Company-Id` y el usuario autenticado esta limitado a una sola subcompania, el backend puede completar `scopeKey: "subCompanyId"` y `scopeId` automaticamente cuando se omiten. Si tiene varias subcompanias permitidas, el body debe indicar `scopeId`.

### Response `201`

Devuelve la lista completa de accesos activos del usuario.

## Reemplazar accesos

`PUT /api/users/:id/access`

Requiere `users.access.assign`.

Reemplaza todos los accesos del usuario en una transaccion. El arreglo `accesses` puede estar vacio para dejar al usuario sin accesos.

Si se envia `X-Company-Id`, reemplaza solo los accesos del usuario dentro de esa compania. No modifica accesos globales ni accesos de otras companias.

### Body

```json
{
    "accesses": [
        {
            "roleId": "768684d0-5892-442c-b52f-ad7d0ef372c6",
            "companyId": "c118d5f0-55c7-48f0-a17f-784909a61f97"
        }
    ]
}
```

### Response `200`

Devuelve `UserAccess[]`.

## Eliminar acceso

`DELETE /api/users/:id/access/:accessId`

Requiere `users.access.assign`.

Si se envia `X-Company-Id`, solo puede eliminar un acceso de esa compania.

### Path params

| Parametro  | Tipo   | Descripcion               |
| ---------- | ------ | ------------------------- |
| `id`       | `uuid` | ID del usuario.           |
| `accessId` | `uuid` | ID del acceso a eliminar. |

### Response `200`

```json
{
    "id": "2db0b73a-b3cb-49a6-8fdf-f81789ab2117",
    "deleted": true
}
```

## Listar permisos efectivos

`GET /api/users/:id/permissions`

Requiere `users.permissions.read`.

Devuelve permisos efectivos calculados desde los roles asignados al usuario. Los permisos se devuelven sin duplicados y ordenados por `code`.

Si se envia `X-Company-Id`, devuelve los permisos efectivos del usuario dentro de esa compania.

### Response `200`

Devuelve `Permission[]`.

Requiere `users.password.update`.

| -------------------- | --------- | --------- | -------------------------------------------------------------- |
| `password`           | `string`  | Si        | Minimo 8 caracteres, una minuscula, una mayuscula y un numero. |

## Desvincular 2FA

`PATCH /api/users/:id/2fa/unlink`

Requiere `users.2fa.unlink`.

Desactiva 2FA, limpia secretos pendientes/confirmados y elimina recovery codes del usuario.

### Response `200`

```json
{
    "twoFactorUnlinked": true
}
```

## Reenviar credenciales

`POST /api/users/:id/resend-credentials`

Requiere `users.credentials.resend`.

Genera una contrasena temporal segura, actualiza el usuario con `mustChangePassword: true` y envia un correo de credenciales usando el idioma preferido del usuario.

### Response `201`

```json
{
    "credentialsSent": true
}
```

## Notas para frontend

Este modulo tambien aplica las reglas CRUD de `Frontend Guidelines`: crear con campos visibles/derivados, editar solo campos documentados en `Actualizar usuario`, confirmar eliminacion y refrescar el listado despues de cada mutacion.

El formulario de alta de usuario no debe mapear todos los campos tecnicos del body como controles visibles. Usa estos controles recomendados:

- Nombre completo: `fullName`.
- Usuario: `username`.
- Correo: `email`.
- Estado: `status`, con `active` como valor inicial.
- Rol: `access.roleId`.
- Compania: `access.companyId`, prellenada con la compania seleccionada cuando exista contexto.
- Alcance: control de UI con opciones `Toda la compania` y `Solo una subcompania`.
- Subcompania: select visible solo si Alcance es `Solo una subcompania`.

Campos que el front puede omitir de forma segura en alta:

- `preferredLanguage`: si se omite, backend usa `es`.
- `password`: si se omite, backend genera una contrasena temporal, marca `mustChangePassword: true` y envia correo de bienvenida.
- `access.scopeKey` y `access.scopeId`: omitir cuando el alcance sea toda la compania.

Mapeo de alcance:

- Toda la compania: enviar `access.roleId` y `access.companyId`; no enviar `scopeKey` ni `scopeId`.
- Solo una subcompania: enviar `access.scopeKey: "subCompanyId"` y `access.scopeId` con el UUID de la subcompania seleccionada.

No muestres `scopeKey` ni `scopeId` como campos editables con esos nombres. Son campos tecnicos del backend; en UI deben representarse como `Alcance` y `Subcompania`.

## Errores comunes

| Estado | Caso                                                                                        |
| ------ | ------------------------------------------------------------------------------------------- |
| `400`  | Body, query o path params invalidos; relaciones invalidas al asignar accesos.               |
| `401`  | Request sin autenticacion valida.                                                           |
| `403`  | Usuario autenticado sin el permiso requerido.                                               |
| `404`  | Usuario, acceso o relacion requerida no encontrada en operaciones que validan existencia.   |
| `409`  | Conflictos de unicidad como `username` o `email` duplicados, segun el filtro Prisma global. |
