# Cardcloud

Base path: `/api/cardcloud`

Este modulo concentra Cardcloud en una sola API local:

- Endpoints proxy hacia la API externa de Cardcloud.
- `GET /api/cardcloud` para listar el espejo local de stock.
- `POST /api/cardcloud/sync` para sincronizar el espejo local global de stock.
- Acciones locales para asignar o desasignar stock Cardcloud a subcompanias.

No existe un controller separado para stock. La tabla local se mantiene por sincronizacion y no expone CRUD manual.

## Listar stock local

`GET /api/cardcloud`

Requiere `RequireSystemAccess` y permiso `cardcloud.read-list`.

Lista el espejo local global de stock Cardcloud. No llama a la API externa y no usa `X-Company-Id`. Devuelve todas las tarjetas sincronizadas desde Cardcloud, tengan o no subcompania asignada.

Query params:

| Campo            | Tipo     | Requerido | Descripcion                                      |
| ---------------- | -------- | --------- | ------------------------------------------------ |
| `page`           | `number` | No        | Pagina. Default `1`.                             |
| `limit`          | `number` | No        | Registros por pagina. Maximo `100`.              |
| `search`         | `string` | No        | Busca en id sincronizado del proveedor, `maskedPan`, `clientId` y subcompania. |
| `providerStatus` | `string` | No        | Filtra por el status crudo recibido de Cardcloud. |
| `subCompanyId`   | `string` | No        | Filtra por subcompania asignada.                     |

Response `200`:

```json
{
    "data": [
        {
            "id": "stock-id",
            "subCompanyId": null,
            "assignedCardId": null,
            "maskedPan": "XXXXXXXXXXXX1001",
            "clientId": "CLIENT-001",
            "balance": "1000.00",
            "providerStatus": "blocked",
            "assignedCard": null,
            "subCompany": null
        }
    ],
    "meta": {
        "total": 1,
        "page": 1,
        "limit": 25,
        "totalPages": 1,
        "hasNext": false,
        "hasPrev": false
    }
}
```

Reglas:

- Usa `GET /api/cardcloud?page=1&limit=25&search=` para la tabla global de stock.
- No enviar `X-Company-Id`.
- El listado es completo/global porque representa el espejo de las tarjetas que vienen de Cardcloud externo.
- No permite crear, editar ni borrar datos sincronizados del proveedor.

## Sincronizar espejo local

`POST /api/cardcloud/sync`

Requiere `RequireSystemAccess` y permiso `cardcloud.sync`.

Sincroniza todas las tarjetas disponibles desde Cardcloud hacia el espejo local global de stock. No usa `X-Company-Id` porque todavia no esta asignando operacion a una compania o subcompania.

No sincroniza datos sensibles como CVV, NIP o PAN completo.

Response `200`:

```json
{
    "synced": 20,
    "skipped": 0,
    "removed": 1
}
```

Reglas:

- Si Cardcloud no esta configurado, devuelve `400`.
- `providerStatus` se guarda como texto crudo recibido desde Cardcloud; no se normaliza al enum local.
- `maskedPan` guarda el PAN enmascarado con `X` y deja visibles solo los ultimos 4 digitos de `masked_pan`. Ejemplo: `516152XXXXXX4398` se guarda como `XXXXXXXXXXXX4398`.
- `externalId` se toma solo de `card_id`; el campo externo `card_external_id` no se usa para crear o actualizar el espejo.
- `skipped` cuenta tarjetas sin `card_id` util o duplicadas.
- Las tarjetas que ya no vengan en Cardcloud se marcan como `inactive` en el espejo local.
- El sync actualiza datos del proveedor (`externalId`, `maskedPan`, `clientId`, `balance`, `providerStatus`) sin cambiar la asignacion local a subcompania. `externalId` se conserva solo para sincronizacion interna y no se devuelve en las respuestas del stock local.
- `balance` se guarda cifrado en base de datos y se devuelve descifrado como string decimal.

## Asignacion local de stock

Estas acciones no llaman a la API externa. Solo administran a que subcompania pertenece una tarjeta dentro del espejo local.

`PATCH /api/cardcloud/:id/assign-sub-company`

Requiere `RequireSystemAccess` y permiso `cardcloud.sub-company.assign`.

Body:

| Campo          | Tipo     | Requerido | Descripcion                         |
| -------------- | -------- | --------- | ----------------------------------- |
| `subCompanyId` | `string` | Si        | Subcompania destino del stock local. |

Response `200`:

```json
{
    "id": "stock-id",
    "subCompanyId": "sub-company-id",
    "assignedCardId": null,
    "maskedPan": "XXXXXXXXXXXX1001",
    "clientId": "CLIENT-001",
    "balance": "1000.00",
    "providerStatus": "blocked",
    "assignedCard": null,
    "subCompany": {
        "id": "sub-company-id",
        "companyId": "company-id",
        "key": "DEMO-SUB-01",
        "name": "Demo Sub Company 01"
    }
}
```

Reglas:

- `subCompanyId` debe existir y estar activo.
- No usa `X-Company-Id`; la subcompania determina a que compania queda asociado el stock.
- Si la subcompania no existe o esta inactiva, devuelve `400`.
- Si el stock no existe, devuelve `404`.

`PATCH /api/cardcloud/:id/unassign-sub-company`

Requiere `RequireSystemAccess` y permiso `cardcloud.sub-company.unassign`.

No recibe body.

Response `200`: mismo formato que asignacion, con `subCompanyId: null` y `subCompany: null`.

Reglas:

- No usa `X-Company-Id`.
- Quita la relacion local con subcompania; no modifica la tarjeta en Cardcloud externo.

## Transferencias

`POST /api/cardcloud/account/transfer`

Requiere `RequireSystemAccess`.

Body:

| Campo             | Tipo                            | Requerido | Descripcion         |
| ----------------- | ------------------------------- | --------- | ------------------- |
| `sourceType`      | `account \| subaccount \| card` | Si        | Origen.             |
| `source`          | `string`                        | Si        | ID externo origen.  |
| `destinationType` | `account \| subaccount \| card` | Si        | Destino.            |
| `destination`     | `string`                        | Si        | ID externo destino. |
| `amount`          | `number`                        | Si        | Monto.              |
| `description`     | `string`                        | Si        | Descripcion.        |

Si `amount` es negativo, backend invierte origen/destino y envia el monto absoluto a Cardcloud, igual que BackendGas.

Response `200`: respuesta cruda de Cardcloud para `/v1/transfer`.

`POST /api/cardcloud/account/transfer-bulk`

Requiere `RequireSystemAccess`.

Body:

```json
{
    "transfers": [
        {
            "sourceType": "account",
            "source": "ACCOUNT-ID",
            "destinationType": "card",
            "destination": "CARD-ID",
            "amount": 100,
            "description": "Carga inicial"
        }
    ]
}
```

Response `200`:

```json
{
    "results": [
        {
            "index": 0,
            "success": true,
            "data": {}
        }
    ],
    "summary": {
        "total": 1,
        "succeeded": 1,
        "failed": 0
    }
```

## Reglas y seguridad

- `CardcloudExternalService` es el unico servicio que se conecta a la API externa.
- `CardcloudService` es el servicio local del modulo y coordina proxy, normalizacion de transferencias y sincronizacion del espejo.
- Los endpoints con alcance `Sistema` no usan `X-Company-Id`; requieren acceso global de sistema y permisos `cardcloud.*`.
- `GET /api/cardcloud` no usa `X-Company-Id`; lista el stock local global sincronizado desde Cardcloud.
- `POST /api/cardcloud/sync` no usa `X-Company-Id`; requiere `RequireSystemAccess`.
- Las acciones `assign-sub-company` y `unassign-sub-company` no usan `X-Company-Id`; requieren acceso de sistema.
- No filtres datos sensibles solo en frontend. Los endpoints `sensitive`, `cvv` y `nip` deben mostrarse solo bajo accion explicita, con confirmacion, y nunca persistirse ni loguearse en frontend.

## Notas para frontend

No generes una pagina CRUD nueva por cada endpoint de Cardcloud. Los endpoints externos son acciones tecnicas del proveedor y deben exponerse solo como herramientas puntuales para usuarios globales autorizados.

Para mostrar el stock local, consume `GET /api/cardcloud?page=1&limit=25&search=` sin `X-Company-Id`. Esta tabla es global y puede vivir dentro de tarjetas/stock Cardcloud, pero no debe permitir editar campos sincronizados.

La accion `Sincronizar Cardcloud` debe ser un boton/accion tecnica de administracion global, no una pagina separada ni una accion por compania. Solo mostrarla cuando `capabilities.permissions` incluya `cardcloud.sync` y el usuario tenga acceso de sistema.

Para asignar stock a una subcompania, usa `PATCH /api/cardcloud/:id/assign-sub-company` desde una accion del inventario/stock. El usuario no debe escribir `subCompanyId`; debe seleccionarlo desde un select filtrado por la compania actual.

No crees una pantalla CRUD manual para modificar datos sincronizados como `maskedPan`, `clientId`, `balance` o `providerStatus`. Esos campos vienen de Cardcloud y se refrescan con `sync`; `maskedPan` se muestra enmascarado con `X` y solo los ultimos 4 digitos visibles. Usa `id` para identificar el stock local; no dependas de ids externos en la UI.
