# Cardcloud

Base path: `/api/cardcloud`

Este modulo concentra Cardcloud en una sola API local:

- Endpoints proxy hacia la API externa de Cardcloud.
- `POST /api/cardcloud/sync` para sincronizar el espejo local de stock.

No existe un controller separado para stock. La tabla local se mantiene por sincronizacion y no expone CRUD manual.

## Sincronizar espejo local

`POST /api/cardcloud/sync`

Requiere `cardcloud.sync` y header `X-Company-Id`.

Sincroniza tarjetas desde Cardcloud hacia el espejo local de stock para la compania seleccionada. Si una tarjeta externa coincide con `Card.externalId` de una tarjeta activa de esa compania, queda relacionada en `assignedCardId`.

Para evitar mezclar clientes, una tarjeta solo entra al espejo de la compania seleccionada cuando cumple al menos una condicion:

- Coincide con una `Card.externalId` activa de la compania.
- No tiene tarjeta local asignada, pero `client_id` de Cardcloud coincide con `Company.externalId`.

No sincroniza datos sensibles como CVV, NIP o PAN completo.

Response `200`:

```json
{
    "synced": 20,
    "skipped": 0,
    "removed": 1
}
```

Reglas:

- Si Cardcloud no esta configurado, devuelve `400`.
- Si el usuario esta limitado a subcompanias, devuelve `400`; la sincronizacion requiere alcance completo de compania.
- `providerStatus` se normaliza a `active`, `inactive` o `suspended`.
- `skipped` cuenta tarjetas sin ID externo util, duplicadas o que no pertenecen a la compania seleccionada.
- Las tarjetas que ya no vengan en Cardcloud se marcan como `inactive` dentro de la compania seleccionada.

## Cuenta

`GET /api/cardcloud/account`

Requiere `RequireSystemAccess`.

Response `200`: respuesta cruda de Cardcloud para `/v1/account`.

`GET /api/cardcloud/account/movements`

Requiere `RequireSystemAccess`.

Query params opcionales:

| Campo  | Tipo     | Descripcion                       |
| ------ | -------- | --------------------------------- |
| `from` | `string` | Timestamp Unix inicial, segundos. |
| `to`   | `string` | Timestamp Unix final, segundos.   |

Response `200`: respuesta cruda de Cardcloud para `/v1/account/movements`.

## Transferencias

`POST /api/cardcloud/account/transfer`

Requiere `RequireSystemAccess`.

Body:

| Campo             | Tipo                            | Requerido | Descripcion         |
| ----------------- | ------------------------------- | --------- | ------------------- |
| `sourceType`      | `account \| subaccount \| card` | Si        | Origen.             |
| `source`          | `string`                        | Si        | ID externo origen.  |
| `destinationType` | `account \| subaccount \| card` | Si        | Destino.            |
| `destination`     | `string`                        | Si        | ID externo destino. |
| `amount`          | `number`                        | Si        | Monto.              |
| `description`     | `string`                        | Si        | Descripcion.        |

Si `amount` es negativo, backend invierte origen/destino y envia el monto absoluto a Cardcloud, igual que BackendGas.

Response `200`: respuesta cruda de Cardcloud para `/v1/transfer`.

`POST /api/cardcloud/account/transfer-bulk`

Requiere `RequireSystemAccess`.

Body:

```json
{
    "transfers": [
        {
            "sourceType": "account",
            "source": "ACCOUNT-ID",
            "destinationType": "card",
            "destination": "CARD-ID",
            "amount": 100,
            "description": "Carga inicial"
        }
    ]
}
```

Response `200`:

```json
{
    "results": [
        {
            "index": 0,
            "success": true,
            "data": {}
        }
    ],
    "summary": {
        "total": 1,
        "succeeded": 1,
        "failed": 0
    }
}
```

## Reglas y seguridad

- `CardcloudExternalService` es el unico servicio que se conecta a la API externa.
- `CardcloudService` es el servicio local del modulo y coordina proxy, normalizacion de transferencias y sincronizacion del espejo.
- Los endpoints con alcance `Sistema` no usan `X-Company-Id`; requieren acceso global de sistema y permisos `cardcloud.*`.
- `POST /api/cardcloud/sync` si usa `X-Company-Id`; requiere permiso `cardcloud.sync` en la compania seleccionada.
- No filtres datos sensibles solo en frontend. Los endpoints `sensitive`, `cvv` y `nip` deben mostrarse solo bajo accion explicita, con confirmacion, y nunca persistirse ni loguearse en frontend.

## Notas para frontend

No generes una pagina CRUD nueva por cada endpoint de Cardcloud. Los endpoints externos son acciones tecnicas del proveedor y deben exponerse solo como herramientas puntuales para usuarios globales autorizados.

La accion `Sincronizar Cardcloud` debe ser un boton/accion de la pantalla de tarjetas o administracion Cardcloud, no una pagina separada. Solo mostrarla cuando `capabilities.permissions` incluya `cardcloud.sync`.

Para que el stock no asignado de Cardcloud aparezca en una compania, `Company.externalId` debe guardar el mismo valor que `client_id` en Cardcloud. Si no se configura, el sync solo importara tarjetas que ya coincidan con una `Card.externalId` local de esa compania.
