title: Permissions API

## Base

Base path: `/api/permissions`

Todos los endpoints requieren autenticacion y el permiso indicado en cada operacion. Los permisos se asignan a roles desde `/api/roles/:id/permissions` y los guards validan sus `code`.

Esta API es de administracion global. Los permisos `permissions.*` deben venir de un acceso global (`companyId: null`).

## Modelos

### Permission

```json
{
    "id": "9ccbb5e9-5184-4125-9d9c-cb43e9df6f00",
    "code": "users.read-list",
    "name": "Listar usuarios",
    "description": "Permite consultar el listado de usuarios."
}
```

| Campo         | Tipo             | Descripcion                                                           |
| ------------- | ---------------- | --------------------------------------------------------------------- |
| `id`          | `string`         | UUID v4 del permiso.                                                  |
| `code`        | `string`         | Codigo unico del permiso; se normaliza a minusculas en create/update. |
| `name`        | `string \| null` | Nombre visible opcional.                                              |
| `description` | `string \| null` | Descripcion opcional.                                                 |

### PaginatedPermissionResponse

```json
{
    "data": [],
    "meta": {
        "total": 0,
        "page": 1,
        "limit": 10,
        "totalPages": 1,
        "hasNext": false,
        "hasPrev": false
    }
}
```

## Endpoints

| Metodo   | Ruta                   | Permiso                 | Descripcion                               |
| -------- | ---------------------- | ----------------------- | ----------------------------------------- |
| `POST`   | `/api/permissions`     | `permissions.create`    | Crea un permiso.                          |
| `GET`    | `/api/permissions`     | `permissions.read-list` | Lista permisos con paginacion y busqueda. |
| `GET`    | `/api/permissions/:id` | `permissions.read-one`  | Obtiene un permiso por ID.                |
| `PATCH`  | `/api/permissions/:id` | `permissions.update`    | Actualiza un permiso.                     |
| `DELETE` | `/api/permissions/:id` | `permissions.delete`    | Elimina un permiso.                       |

## Crear permiso

`POST /api/permissions`

Requiere `permissions.create`.

### Body

```json
{
    "code": "users.read-list",
    "name": "Listar usuarios",
    "description": "Permite consultar el listado de usuarios."
}
```

| Campo         | Tipo             | Requerido | Reglas                                                 |
| ------------- | ---------------- | --------- | ------------------------------------------------------ |
| `code`        | `string`         | Si        | Se recorta, colapsa espacios y normaliza a minusculas. |
| `name`        | `string \| null` | No        | String vacio se transforma a `null`.                   |
| `description` | `string \| null` | No        | String vacio se transforma a `null`.                   |

### Response `201`

Devuelve `Permission`.

## Listar permisos

`GET /api/permissions`

Requiere `permissions.read-list`.

### Query params

| Parametro | Tipo      | Reglas               | Descripcion                                                                          |
| --------- | --------- | -------------------- | ------------------------------------------------------------------------------------ |
| `page`    | `number`  | Entero >= 1          | Pagina solicitada; por defecto `1`.                                                  |
| `limit`   | `number`  | Entero entre 1 y 100 | Registros por pagina; por defecto `10`.                                              |
| `search`  | `string`  | Opcional             | Busca por `code` o `name`, sin distinguir mayusculas.                                |
| `all`     | `boolean` | Opcional             | Si es `true`, omite el offset y solicita hasta 100 registros si no se envia `limit`. |

### Example

```http
GET /api/permissions?page=1&limit=20&search=users
```

### Response `200`

Devuelve `PaginatedPermissionResponse`.

## Obtener permiso

`GET /api/permissions/:id`

Requiere `permissions.read-one`.

### Path params

| Parametro | Tipo   | Descripcion     |
| --------- | ------ | --------------- |
| `id`      | `uuid` | ID del permiso. |

### Response `200`

Devuelve `Permission`. Comportamiento actual: si el permiso no existe, devuelve `null`.

## Actualizar permiso

`PATCH /api/permissions/:id`

Requiere `permissions.update`.

### Body

Todos los campos son opcionales.

```json
{
    "code": "users.read-list",
    "name": "Listar usuarios",
    "description": "Permite consultar el listado de usuarios."
}
```

| Campo         | Tipo             | Reglas                                                 |
| ------------- | ---------------- | ------------------------------------------------------ |
| `code`        | `string`         | Se recorta, colapsa espacios y normaliza a minusculas. |
| `name`        | `string \| null` | String vacio se transforma a `null`.                   |
| `description` | `string \| null` | String vacio se transforma a `null`.                   |

### Response `200`

Devuelve `Permission`.

## Eliminar permiso

`DELETE /api/permissions/:id`

Requiere `permissions.delete`.

### Response `200`

```json
{
    "id": "9ccbb5e9-5184-4125-9d9c-cb43e9df6f00",
    "deleted": true
}
```

## Uso en roles y usuarios

Los permisos no se asignan directamente a usuarios. El flujo actual es:

1. Crear o consultar permisos en `/api/permissions`.
2. Asignar permisos a un rol con `PUT /api/roles/:id/permissions`.
3. Asignar el rol a un usuario desde `/api/users/:id/access`.
4. Consultar permisos efectivos del usuario en `/api/users/:id/permissions`.

## Notas para frontend

Este modulo tambien aplica las reglas CRUD de `Frontend Guidelines`: crear con campos visibles/derivados, editar solo campos documentados en `Actualizar permiso`, confirmar eliminacion y refrescar el listado despues de cada mutacion.

Esta pantalla debe tratarse como administracion global y normalmente solo debe mostrarse a super administradores. No envies `X-Company-Id` en estos endpoints.

Campos recomendados para formularios:

- Codigo: `code`, valor tecnico que usan guards y backend.
- Nombre: `name`, etiqueta legible para administradores.
- Descripcion: `description`, opcional.

El front no debe generar permisos dinamicos por modulo ni permitir que usuarios de compania creen permisos. El catalogo de permisos es fuente de verdad del backend y se asigna a roles.

Esta pagina es opcional para el producto final y normalmente solo se muestra a super administradores. Si el front no expone mantenimiento tecnico de RBAC, usa `GET /api/permissions` solo como fuente para el selector de permisos dentro de roles.

En pantallas de usuarios, no muestres permisos como controles editables; muestra roles. Los permisos efectivos pueden mostrarse solo como lectura avanzada o diagnostico.

## Errores comunes

| Estado | Caso                                                                         |
| ------ | ---------------------------------------------------------------------------- |
| `400`  | Body, query o path params invalidos.                                         |
| `401`  | Request sin autenticacion valida.                                            |
| `403`  | Usuario autenticado sin el permiso requerido.                                |
| `404`  | Permiso no encontrado en operaciones de update o delete.                     |
| `409`  | Conflictos de unicidad como `code` duplicado, segun el filtro Prisma global. |
