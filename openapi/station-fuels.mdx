## Base

Base path: `/api/station-fuels`

Todos los endpoints requieren autenticacion y permisos `station-fuels.*`. El borrado desactiva el registro cambiando `status` a `inactive`.

Este es un modulo tecnico de relacion entre estaciones y combustibles. No debe renderizarse como pagina principal del frontend ni como opcion de menu; debe consumirse dentro del detalle o edicion de una estacion.

## Alcance por compania

Requiere header `X-Company-Id: <companyId>`. Todas las operaciones quedan limitadas a esa compania. Si el acceso del usuario tiene `scopeKey: "subCompanyId"`, `stationId` y la relacion consultada tambien deben pertenecer a una de esas subcompanias.

## Modelo

```json
{
    "id": "352ae0cd-6f6c-4f9a-8943-b5fc6a973b21",
    "stationId": "48c478d5-d9a6-4a8a-a98f-6cc0a3ea7d3f",
    "fuelId": "e9ef8b38-c983-4b9c-a51c-7d39ff7b1c84",
    "status": "active",
    "station": {
        "id": "48c478d5-d9a6-4a8a-a98f-6cc0a3ea7d3f",
        "stationNumber": "ST-001",
        "name": "Estacion Centro"
    },
    "fuel": {
        "id": "e9ef8b38-c983-4b9c-a51c-7d39ff7b1c84",
        "code": "DIESEL",
        "name": "Diesel"
    }
}
```

Las relaciones se devuelven como summaries controlados: `station` y `fuel`.

## Endpoints

| Metodo   | Ruta                     | Permiso                   | Descripcion                         |
| -------- | ------------------------ | ------------------------- | ----------------------------------- |
| `POST`   | `/api/station-fuels`     | `station-fuels.create`    | Crea relacion estacion-combustible. |
| `GET`    | `/api/station-fuels`     | `station-fuels.read-list` | Lista relaciones.                   |
| `GET`    | `/api/station-fuels/:id` | `station-fuels.read-one`  | Obtiene una relacion.               |
| `PATCH`  | `/api/station-fuels/:id` | `station-fuels.update`    | Actualiza estado de la relacion.    |
| `DELETE` | `/api/station-fuels/:id` | `station-fuels.delete`    | Desactiva una relacion.             |

## Listado

`GET /api/station-fuels`

Query params: `page`, `limit`, `all`, `status`, `stationId`, `fuelId`.

Este listado no tiene busqueda textual propia.

Para columnas visibles, usa `station.name` y `fuel.name`; conserva IDs para filtros y acciones.

Response `200`: resultado paginado de `StationFuel`.

## Crear

`POST /api/station-fuels`

```json
{
    "stationId": "48c478d5-d9a6-4a8a-a98f-6cc0a3ea7d3f",
    "fuelId": "e9ef8b38-c983-4b9c-a51c-7d39ff7b1c84",
    "status": "active"
}
```

Requeridos: `stationId`, `fuelId`. `stationId` debe estar activo y pertenecer a la compania enviada en `X-Company-Id`. `fuelId` debe estar activo.

Opcional: `status`.

Si ya existe una relacion con el mismo `stationId` y `fuelId`, el backend no crea una fila nueva. Reactiva o actualiza la relacion existente con el `status` enviado, usando `active` por defecto. Esto permite volver a agregar desde el tab de combustibles una relacion que antes fue desactivada.

Response `201`: `StationFuel`.

## Obtener

`GET /api/station-fuels/:id`

Path params: `id` UUID de la relacion estacion-combustible.

Response `200`: `StationFuel`.

## Actualizar

`PATCH /api/station-fuels/:id`

```json
{
    "status": "inactive"
}
```

Solo permite actualizar `status`.

Response `200`: `StationFuel` actualizada.

## Desactivar

`DELETE /api/station-fuels/:id`

Response `200`:

```json
{
    "id": "352ae0cd-6f6c-4f9a-8943-b5fc6a973b21",
    "status": "inactive"
}
```

## Notas para frontend

Este modulo tambien aplica las reglas CRUD de `Frontend Guidelines`: crear con campos visibles/derivados, editar solo campos documentados en `Actualizar`, confirmar desactivacion y refrescar el listado despues de cada mutacion.

El front debe enviar `X-Company-Id`. `stationId` debe venir de estaciones activas visibles para la compania y alcance seleccionados; `fuelId` viene del catalogo global de combustibles activos.

Este modulo normalmente se consume como seccion hija de una estacion: `Combustibles disponibles`. Para agregar un combustible a una estacion, crea la relacion con `stationId` y `fuelId`; para quitarlo de la vista operativa, desactiva o elimina la relacion segun la accion disponible en UI.

No generes una pantalla CRUD independiente para `station-fuels`. El usuario debe llegar a estas acciones desde `Estaciones > Detalle/Editar > Combustibles disponibles`.

Si el usuario vuelve a agregar un combustible que habia quitado de la estacion, llama nuevamente `POST /api/station-fuels` con el mismo `stationId` y `fuelId`; no intentes crear una variante local ni cambiar IDs. El backend reactivara la relacion existente.

Campos recomendados para formularios:

- Estacion: `stationId`.
- Combustible: `fuelId`.
- Estado: `status`, con `active` como valor inicial.

En tablas usa `station.name` y `fuel.name`; no muestres `stationId` ni `fuelId` como columnas principales. Este modulo no requiere busqueda textual, solo filtros por estacion, combustible y estado.

## Errores comunes

| Estado | Caso                                             |
| ------ | ------------------------------------------------ |
| `400`  | Relacion invalida o parametros invalidos.        |
| `401`  | Request sin autenticacion valida.                |
| `403`  | Usuario autenticado sin el permiso requerido.    |
| `404`  | Relacion estacion-combustible no encontrada.     |
| `409`  | Conflictos de unicidad segun constraints Prisma. |
